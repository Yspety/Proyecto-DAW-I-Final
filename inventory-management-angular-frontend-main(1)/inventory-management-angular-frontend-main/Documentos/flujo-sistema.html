🗺 01 · Visión General del Sistema

El sistema está compuesto por dos aplicaciones independientes que se comunican entre sí. El frontend Angular es la interfaz visual que usa el usuario; el backend Spring Boot almacena y protege los datos. Ambos se ejecutan en paralelo en la misma máquina durante desarrollo.

🖥 Frontend Angular 18

Puerto 4200

Componentes standalone, Signals, Tailwind CSS. El navegador descarga este código y lo ejecuta. Nunca toca la base de datos directamente.

⚙️ Backend Spring Boot 3.5

Puerto 8080

API REST con Spring Security. Expone endpoints /api/** protegidos por HTTP Basic Auth. Lee y escribe en MySQL.

💡
¿Cómo se comunican? Durante el desarrollo, el servidor Angular tiene un proxy (proxy.conf.js) que redirige todas las peticiones que empiecen con /api/ al backend en http://localhost:8080. Así el navegador siempre habla con el puerto 4200 y Angular maneja la redirección.

¿Qué ocurre técnicamente al hacer clic en "Iniciar sesión"?

1
Validación del formulario reactivo

El LoginComponent usa ReactiveFormsModule de Angular. Los campos usuario y contraseña tienen validadores Validators.required. Si alguno está vacío, el botón no ejecuta la acción y se muestran los errores.

2
Codificación Base64 de las credenciales

El método AuthService.login() convierte las credenciales a formato HTTP Basic Auth: toma la cadena "usuario:password" y la transforma con btoa() a Base64. Ej: "admin:1234" → "YWRtaW46MTIzNA==".

3
Almacenamiento del token en localStorage

El token Base64 se guarda en localStorage bajo la clave erp_token. Esto es temporal y persiste aunque el usuario recargue el navegador, hasta que haga logout.

4
Verificación contra el backend: GET /api/usuarios/me

El servicio realiza una petición HTTP con el header Authorization: Basic YWRtaW46MTIzNA== al endpoint GET /api/usuarios/me. Si las credenciales son correctas, Spring Boot responde con los datos del usuario (nombre, apellido, rol). Si son incorrectas, responde con HTTP 401 Unauthorized.

5
Almacenamiento del perfil del usuario

Si el backend responde con éxito, el AuthService guarda el nombre completo y el rol en localStorage('erp_user') como JSON. Este dato se usa en el Navbar para mostrar el nombre del usuario y en el guard para decidir si mostrar el menú de administración.

6
Navegación a /home

Una vez almacenados token y perfil, el router navega automáticamente a /home. Si las credenciales eran incorrectas, se limpia el localStorage y se muestra el mensaje "Credenciales inválidas" en el formulario.

Código clave del login

// auth.service.ts
login(usuario: string, password: string): Observable<void> {
  const token = btoa(`${usuario}:${password}`); // Base64
  localStorage.setItem('erp_token', token);

  return this.http.get<ApiEnvelope<Usuario>>('/api/usuarios/me').pipe(
    map(resp => {
      const user = resp.data;
      localStorage.setItem('erp_user', JSON.stringify({
        nombreCompleto: `${user.nombre} ${user.apellido}`,
        rol: user.rolDescripcion
      }));
    }),
    catchError(() => {
      localStorage.removeItem('erp_token');
      return throwError(() => new Error('Credenciales inválidas'));
    })
  );
}
⚠️
Sobre la seguridad de HTTP Basic Auth El token Base64 en localStorage NO es cifrado; es solo una codificación. En producción real, esto se combina con HTTPS para que el header viaje cifrado por TLS. Para esta aplicación académica es suficiente, pero en entornos productivos se recomienda JWT con HTTPS.

🏠 03 · Navegación Post-Login

Tras el login exitoso, el usuario llega al Dashboard (Home). Cada vez que navega a una ruta protegida, el authGuard verifica que localStorage('erp_token') exista. Si alguien intenta acceder a /productos directamente sin token, es redirigido a /login.

🛡
authGuard — El portero de las rutas Es una función (CanActivateFn) registrada en cada ruta protegida de app.routes.ts. Revisa si hay token en localStorage. Si no hay, redirige a /login. No valida si el token es correcto en el backend — esa validación ocurre en cada petición HTTP mediante el interceptor.

El interceptor: el "cartero" automático

El BasicAuthInterceptor está registrado globalmente en app.config.ts. Intercepta todas las peticiones HTTP de la aplicación antes de que salgan al servidor. Si la URL contiene /api/, añade automáticamente el header de autenticación. El desarrollador no tiene que escribir el header en cada servicio.

// basic-auth.interceptor.ts
export const basicAuthInterceptor: HttpInterceptorFn = (req, next) => {
  const token = localStorage.getItem('erp_token');

  if (token && req.url.includes('/api/')) {
    const authReq = req.clone({
      setHeaders: { Authorization: `Basic ${token}` }
    });
    return next(authReq);  // envía la petición modificada
  }
  return next(req);  // sin token, la envía igual (solo /login no tiene token)
};

📦 04 · CRUD Completo: Ejemplo con Productos

La sección de Productos es el ejemplo perfecto del patrón CRUD implementado en todo el sistema. Veamos el flujo completo paso a paso con un producto real.

🎯
Producto de ejemplo: Aceite de cocina Sol 1L SKU: PRD-001 · Categoría: Alimentos · Unidad: Litros · Precio: S/ 8.50

📋 R — Read: Listar productos al entrar a la página

1
Angular ejecuta ngOnInit del componente

Al navegar a /productos, Angular instancia ProductosComponent. En ngOnInit() se llama cargar(), que activa la señal cargando = signal(true) para mostrar un spinner al usuario.

2
forkJoin lanza 3 peticiones en paralelo

El componente necesita datos de 3 endpoints simultáneamente. Usa forkJoin() de RxJS para ejecutar las 3 llamadas al mismo tiempo y esperar a que todas terminen. Esto es más eficiente que hacerlas en secuencia.

3
El interceptor añade el header de autenticación

Antes de que las peticiones salgan, el BasicAuthInterceptor las intercepta y añade Authorization: Basic YWRtaW46MTIzNA==. El backend valida este header con Spring Security en cada petición.

4
Los servicios reciben y "desenvuelven" la respuesta

El backend responde con el envelope { success, message, data, total }. Cada servicio usa .pipe(map(r => r.data)) para extraer solo el array de datos antes de devolverlo al componente.

5
El componente renderiza la tabla

Los datos se guardan en variables locales del componente. Angular actualiza el DOM con la directiva *ngFor para renderizar cada fila. Se desactiva la señal cargando = signal(false).

// productos.component.ts — carga inicial
ngOnInit() {
  this.cargar();
}

cargar() {
  this.cargando.set(true);
  forkJoin({
    productos:  this.productoService.getAll(),   // GET /api/productos
    categorias: this.categoriaService.getAll(),  // GET /api/categorias
    unidades:   this.unidadService.getAll()      // GET /api/unidades
  }).subscribe({
    next: ({ productos, categorias, unidades }) => {
      this.productos  = productos;
      this.filtrados  = productos;  // copia para filtrar localmente
      this.categorias = categorias;
      this.unidades   = unidades;
      this.cargando.set(false);
    }
  });
}

➕ C — Create: Registrar un nuevo producto

1
El usuario hace clic en "+ Nuevo Producto"

Se activan las señales modalAbierto = signal(true) y modoEdicion = signal(false). El formulario modal aparece en pantalla vacío.

2
El usuario completa el formulario

El formulario usa FormBuilder con validaciones. Para nuestro ejemplo: SKU=PRD-004, Nombre=Leche Gloria 1L, Categoría=Lácteos, Unidad=Litros, Precio=3.80.

3
Guardar activa la señal de "cargando botón"

Al hacer clic en "Guardar", se activa guardando = signal(true) para deshabilitar el botón y evitar doble envío. El formulario es inválido mientras guardando es true.

4
POST /api/productos con el body JSON

El servicio ProductoService.create() envía los datos. El interceptor añade el header de auth. Spring Boot valida, guarda en MySQL y devuelve el objeto creado con su ID asignado.

5
Se cierra el modal y se recarga la lista

En el callback next() de la suscripción: se llama cargar() para obtener la lista actualizada, se cierra el modal con modalAbierto.set(false) y se muestra la alerta de éxito.

// productos.component.ts — guardar producto
guardar() {
  if (this.form.invalid) return;
  this.guardando.set(true);

  const body: ProductoRequest = this.form.value;
  const accion = this.modoEdicion()
    ? this.productoService.update(this.idEditando!, body)  // PUT
    : this.productoService.create(body);                   // POST

  accion.subscribe({
    next: () => {
      this.alerta.set({ tipo: 'ok', msg: 'Producto guardado correctamente' });
      this.modalAbierto.set(false);
      this.cargar();  // refresca la lista
    },
    error: () => this.alerta.set({ tipo: 'err', msg: 'Error al guardar el producto' }),
    complete: () => this.guardando.set(false)
  });
}

✏️ U — Update: Editar un producto existente

Al hacer clic en el botón "Editar" de una fila, el componente ejecuta:

  1. Activa modoEdicion = signal(true) y guarda el ID del producto en idEditando.
  2. Parchea el formulario con los datos actuales del producto usando form.patchValue(producto).
  3. Abre el modal con los campos pre-cargados.
  4. Al guardar, el servicio ejecuta PUT /api/productos/{id} con el body actualizado.
  5. El backend actualiza el registro en MySQL y devuelve el objeto modificado.

🗑 D — Delete: Eliminar un producto

1
Confirmación del usuario

Al hacer clic en "Eliminar", el componente ejecuta confirm('¿Seguro que deseas eliminar este producto?'). Si el usuario cancela, no se ejecuta nada.

2
DELETE /api/productos/{id}

Si confirma, el servicio ejecuta ProductoService.delete(id) que lanza DELETE /api/productos/4. Spring Boot elimina el registro y responde con HTTP 200.

3
Recarga y alerta

Igual que en create/update: se llama cargar() y se muestra la alerta de éxito. Si el backend rechaza la eliminación (por integridad referencial), se muestra la alerta de error.

🔍 Filtrado local por texto

La búsqueda es instantánea y sin peticiones al backend. Al escribir en el campo de búsqueda, el componente filtra el array productos (ya cargado en memoria) y actualiza filtrados:

filtrar(termino: string) {
  const q = termino.toLowerCase();
  this.filtrados = this.productos.filter(p =>
    p.nombre.toLowerCase().includes(q) ||
    p.sku.toLowerCase().includes(q)   ||
    p.categoriaNombre.toLowerCase().includes(q)
  );
}

🔔 05 · Sistema de Alertas

Cada componente tiene una señal alerta = signal<{tipo, msg} | null>(null). Cuando se completa una operación, se activa con un mensaje. La alerta desaparece automáticamente tras 5 segundos.

✅ Producto guardado correctamente

Se muestra cuando el backend responde con HTTP 200/201. Clase Tailwind: erp-alert-success

❌ Error al eliminar el producto

Se muestra cuando el backend responde con HTTP 4xx/5xx. Clase Tailwind: erp-alert-danger

// auto-dismiss a los 5 segundos
mostrarAlerta(tipo: 'ok' | 'err', msg: string) {
  this.alerta.set({ tipo, msg });
  setTimeout(() => this.alerta.set(null), 5000);
}

🚪 06 · Cierre de Sesión (Logout)

1
Usuario hace clic en "Cerrar sesión" en el Navbar

El NavbarComponent llama a AuthService.logout(). No se hace ninguna petición al backend.

2
Se limpia el localStorage

Se eliminan localStorage.removeItem('erp_token') y localStorage.removeItem('erp_user'). Después de esto, el token Base64 desaparece de la memoria del navegador.

3
Redirección a /login

El router.navigate(['/login']) lleva al usuario de vuelta a la pantalla de inicio. Si intenta usar el botón "Atrás" del navegador, el authGuard detectará que no hay token y volverá a redirigirlo a /login.
