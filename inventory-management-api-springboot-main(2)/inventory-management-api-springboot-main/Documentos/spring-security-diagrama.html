🛠 ¿Qué es Spring Security en este proyecto?

Concepto clave

Spring Security es un framework que actúa como guardián de todas las peticiones HTTP antes de que lleguen a los controllers. Intercepta cada request, verifica si el usuario tiene credenciales válidas y, solo si pasa la verificación, permite que la petición continúe. En este proyecto se usan dos cadenas de filtros independientes: una para la API REST y otra para la interfaz web MVC.

🔗 Los Dos Filter Chains (SecurityConfig.java)

La clase SecurityConfig.java define dos cadenas de seguridad con @Order. Spring Security evalúa la orden de menor a mayor y usa la primera que coincida con la ruta del request.

@Order(1) — Primero
🔴 API Filter Chain
Ruta que cubre/api/**
Tipo de authHTTP Basic Auth
SesiónSTATELESS (sin sesión)
CSRFDeshabilitado
Acceso sin authNinguno → 401
UsoFrontend Angular, Postman, cURL
@Order(2) — Segundo (comentado)
🔵 MVC Filter Chain
Ruta que cubre/** (todo lo demás)
Tipo de authForm Login (/login)
SesiónSTATEFUL (con sesión HTTP)
CSRFHabilitado (via Thymeleaf)
Acceso sin auth/login, /css/**, /js/** permitidos
UsoInterfaz web Thymeleaf (navegador)
Nota: MVC Chain comentada

En el código actual, el MVC Filter Chain está comentado (/* ... */). Esto significa que actualmente solo el API REST está protegido con Spring Security. La interfaz web MVC no aplica el form-login por código, aunque el código está preparado para activarse cuando se necesite.

🔄 Flujo Completo: HTTP Basic Auth paso a paso

Ejemplo real: el frontend Angular hace GET /api/productos con credenciales admin:admin. Aquí está lo que sucede dentro de Spring Security, línea por línea en cada clase.

1
Cliente (Angular) envía la petición HTTP
El frontend construye el header Authorization codificando las credenciales en Base64. La cadena admin:admin codificada en Base64 es YWRtaW46YWRtaW4=. Este header se agrega automáticamente por el BasicAuthInterceptor de Angular en cada request.
// HTTP Request que sale del frontend Angular GET /api/productos HTTP/1.1 Host: localhost:8080 Authorization: Basic YWRtaW46YWRtaW4= ← Base64("admin:admin") Accept: application/json Content-Type: application/json // ¿Cómo se forma YWRtaW46YWRtaW4=? // btoa("admin:admin") → "YWRtaW46YWRtaW4=" (función JS) // Base64.encode("admin:admin") → "YWRtaW46YWRtaW4=" (Java)
2
FilterChainProxy selecciona el chain correcto
El componente central de Spring Security, FilterChainProxy, recibe el request. Evalúa qué cadena aplica: la ruta es /api/productos, que coincide con el patrón /api/** del chain de @Order(1). Usa ese chain y no evalúa el @Order(2).
// SecurityConfig.java — Spring selecciona este bean porque la ruta es /api/** @Bean @Order(1) public SecurityFilterChain apiFilterChain(HttpSecurity http) throws Exception { http .securityMatcher("/api/**") ← ✅ /api/productos coincide .authorizeHttpRequests(auth -> auth .anyRequest().authenticated() ← toda ruta requiere auth ) .httpBasic(Customizer.withDefaults()) ← activa BasicAuthenticationFilter .csrf(csrf -> csrf.disable()) ← no se necesita token CSRF .sessionManagement(sm -> sm .sessionCreationPolicy(SessionCreationPolicy.STATELESS) ← sin sesión ); return http.build(); }
3
BasicAuthenticationFilter extrae y decodifica las credenciales
El filtro BasicAuthenticationFilter (interno de Spring Security) lee el header Authorization, verifica que empiece con "Basic ", y decodifica el Base64 para obtener el usuario y contraseña separados por :.
// Lógica interna de BasicAuthenticationFilter (Spring Security lo hace automáticamente) String header = request.getHeader("Authorization"); // header = "Basic YWRtaW46YWRtaW4=" String base64Credentials = header.substring(6); ← quita "Basic " // base64Credentials = "YWRtaW46YWRtaW4=" String decoded = new String(Base64.decode(base64Credentials)); // decoded = "admin:admin" String[] parts = decoded.split(":", 2); String username = parts[0]; // "admin" String password = parts[1]; // "admin" // Crea el token de autenticación (aún NO autenticado — solo lleva las credenciales) UsernamePasswordAuthenticationToken authToken = new UsernamePasswordAuthenticationToken(username, password); // Pasa el token al AuthenticationManager para que lo verifique Authentication result = authenticationManager.authenticate(authToken);
4
DaoAuthenticationProvider llama a UserDetailsServiceImpl
El AuthenticationManager delega en DaoAuthenticationProvider (implementación por defecto de Spring Security). Este provider llama a loadUserByUsername("admin") de nuestra clase UserDetailsServiceImpl para buscar el usuario en la base de datos.
// UserDetailsServiceImpl.java — nuestra implementación @Service @RequiredArgsConstructor public class UserDetailsServiceImpl implements UserDetailsService { private final UsuarioRepository usuarioRepository; @Override @Transactional(readOnly = true) ← abre transacción JPA public UserDetails loadUserByUsername(String username) throws UsernameNotFoundException { // Busca el usuario en la tabla "Usuario" por campo "usuario" Usuario usuario = usuarioRepository.findByUsuario(username) .orElseThrow(() -> new UsernameNotFoundException("Usuario no encontrado: " + username) ); // Si no existe → Spring Security retorna 401 automáticamente return new UsuarioPrincipal(usuario); ← crea el principal dentro de @Transactional } }
4b
JPA ejecuta la consulta SQL en MySQL
El método usuarioRepository.findByUsuario("admin") hace que Spring Data JPA genere y ejecute automáticamente la siguiente consulta SQL en la base de datos erp_productos:
-- SQL generado por Spring Data JPA (UsuarioRepository) SELECT u.idusu, u.nombre, u.apellido, u.usuario, u.password, ← el hash BCrypt, ej: "$2a$10$Xyz..." u.activo, u.creado_en, u.idrol, r.idrol, r.descripcion ← "ADMIN" o "ALMACENERO" FROM Usuario u INNER JOIN Rol r ON u.idrol = r.idrol WHERE u.usuario = 'admin' LIMIT 1; -- Resultado que devuelve la DB: -- idusu=1, nombre="Juan", apellido="Pérez", usuario="admin" -- password="$2a$10$hKZdsMlFkdjhQ2kJMO1...", activo=1, idrol=1, descripcion="ADMIN"
5
UsuarioPrincipal se construye (dentro de la transacción JPA)
El constructor de UsuarioPrincipal extrae todos los campos del entity Usuario inmediatamente, mientras la transacción JPA sigue abierta. Esto es crítico: la relación usuario.getRol() es LAZY (se carga bajo demanda). Si se accediera fuera de la transacción lanzaría LazyInitializationException. Al extraer todo en el constructor, se evita ese problema.
// UsuarioPrincipal.java — implementa UserDetails de Spring Security public class UsuarioPrincipal implements UserDetails { private final Integer id; private final String username; private final String password; ← hash BCrypt private final boolean activo; private final String nombre; private final String apellido; private final String rolDescripcion; public UsuarioPrincipal(Usuario usuario) { this.id = usuario.getId(); this.username = usuario.getUsuario(); this.password = usuario.getPassword(); this.activo = usuario.getActivo(); this.nombre = usuario.getNombre(); this.apellido = usuario.getApellido(); this.rolDescripcion = usuario.getRol().getDescripcion(); // ↑ LAZY pero OK: estamos dentro de @Transactional } @Override public Collection<? extends GrantedAuthority> getAuthorities() { // Convierte "ADMIN" → GrantedAuthority "ROLE_ADMIN" return List.of(new SimpleGrantedAuthority("ROLE_" + rolDescripcion)); } @Override public boolean isEnabled() { return activo; } @Override public boolean isAccountNonExpired() { return true; } @Override public boolean isAccountNonLocked() { return true; } @Override public boolean isCredentialsNonExpired(){ return true; } }
6
BCryptPasswordEncoder verifica la contraseña
DaoAuthenticationProvider llama a passwordEncoder.matches() para comparar la contraseña en texto plano "admin" con el hash BCrypt guardado en la base de datos. BCrypt es un algoritmo de hash unidireccional: nunca se puede revertir el hash. En cambio, el algoritmo re-hashea la contraseña ingresada con el mismo "salt" que usó el hash original, y compara los resultados.
// BCryptPasswordEncoder (definido como @Bean en SecurityConfig.java) @Bean public PasswordEncoder passwordEncoder() { return new BCryptPasswordEncoder(); } // Lo que hace DaoAuthenticationProvider internamente: String rawPassword = "admin"; ← lo que escribió el usuario String storedHash = "$2a$10$hKZdsMlFk..."; ← guardado en DB boolean match = passwordEncoder.matches(rawPassword, storedHash); // ↑ BCrypt re-hashea "admin" con el mismo salt del storedHash // Si el resultado coincide → match = true → AUTENTICADO // Si no coincide → match = false → BadCredentialsException → 401 // Anatomía del hash BCrypt: $2a$10$hKZdsMlFkdjhQ2kJMO1xyz... // $2a$ → versión del algoritmo // 10 → "cost factor" (2^10 = 1024 rondas de hashing) // hKZds → salt (22 caracteres, aleatório por registro) // MlFk.. → hash resultado (31 caracteres)
7
SecurityContextHolder almacena la autenticación (sin sesión)
Si BCrypt confirma que la contraseña es correcta, Spring Security almacena el objeto Authentication en el SecurityContextHolder para este thread. Como la cadena API está configurada como STATELESS, no se crea ninguna sesión HTTP. El SecurityContext solo vive durante este request y se limpia al terminar. En el próximo request, el cliente deberá enviar las credenciales nuevamente.
// Spring Security hace esto internamente después de verificar BCrypt: UsernamePasswordAuthenticationToken autenticado = new UsernamePasswordAuthenticationToken( usuarioPrincipal, ← el UsuarioPrincipal que creamos null, ← credenciales se limpian por seguridad usuarioPrincipal.getAuthorities() ← [ROLE_ADMIN] ); SecurityContextHolder.getContext().setAuthentication(autenticado); // Estado del SecurityContext en este momento: // { // authenticated: true // principal: UsuarioPrincipal { id=1, username="admin", rol="ADMIN", ... } // authorities: ["ROLE_ADMIN"] // } // NO se crea JSESSIONID (STATELESS) — cada request repite los pasos 1-7
8
DispatcherServlet enruta al ProductoRestController
Con la autenticación verificada, Spring Security deja pasar el request. El DispatcherServlet lo enruta al método correspondiente del controller. Si el endpoint requiere rol ADMIN (como /api/usuarios), Spring Security verifica que UsuarioPrincipal.getAuthorities() contenga ROLE_ADMIN antes de entrar al método. Para /api/productos cualquier usuario autenticado puede acceder.
// ProductoRestController.java @RestController @RequestMapping("/api/productos") public class ProductoRestController { private final ProductoService productoService; @GetMapping public ResponseEntity<Map<String, Object>> listar() { // Spring Security ya verificó las credenciales — llegamos aquí autenticados List<ProductoResponseDto> productos = productoService.listar(); Map<String, Object> response = new HashMap<>(); response.put("success", true); response.put("message", "Productos listados correctamente"); response.put("data", productos); response.put("total", productos.size()); return ResponseEntity.ok(response); } } // Acceso ADMIN-only (como en UsuarioRestController.java): @GetMapping @PreAuthorize("hasRole('ADMIN')") ← Spring verifica ROLE_ADMIN antes de entrar public ResponseEntity<...> listar() { ... }
9
Respuesta JSON vuelve al cliente
El controller construye el envelope de respuesta y lo serializa a JSON via Jackson. Spring Security limpia el SecurityContext del thread después del response (STATELESS). El cliente Angular recibe el JSON y lo procesa.
// HTTP Response que recibe el frontend Angular HTTP/1.1 200 OK Content-Type: application/json { "success": true, "message": "Productos listados correctamente", "data": [ { "id": 1, "sku": "PRD-001", "nombre": "Arroz Extra", "precioLista": 3.50, "activo": true, // ... más campos } ], "total": 1 }

❌ Casos de Error — ¿Qué pasa si falla la autenticación?

Credenciales incorrectas (contraseña mal)

1
BCryptPasswordEncoder.matches("malpass", hash) → false
2
DaoAuthenticationProvider lanza BadCredentialsException
3
BasicAuthenticationEntryPoint responde con HTTP 401 Unauthorized
4
Header: WWW-Authenticate: Basic realm="Realm"
5
El Controller NO se ejecuta

Usuario no existe en la DB

1
usuarioRepository.findByUsuario("x") → Optional.empty()
2
UserDetailsServiceImpl lanza UsernameNotFoundException
3
Spring Security convierte esto a 401 Unauthorized
4
Por seguridad, el mensaje de error es genérico (no dice si el usuario existe o no)

Sin header Authorization

1
BasicAuthenticationFilter no encuentra el header
2
No hay Authentication en SecurityContext
3
AuthorizationFilter verifica: anyRequest().authenticated() → falla
4
Respuesta: HTTP 401 Unauthorized

Usuario sin rol ADMIN en endpoint restringido

1
Autenticación OK (contraseña correcta)
2
UsuarioPrincipal tiene ROLE_ALMACENERO
3
@PreAuthorize("hasRole('ADMIN')") falla
4
Respuesta: HTTP 403 Forbidden

🔄 PasswordMigrationRunner — Migración automática de contraseñas

¿Para qué sirve?

Cuando se instaló Spring Security, la base de datos tenía contraseñas en texto plano (ej: "admin"). Spring Security no puede comparar texto plano con BCrypt. El PasswordMigrationRunner detecta las contraseñas sin hash al arrancar la aplicación y las migra automáticamente a BCrypt. Es idempotente: la segunda vez que corre, las contraseñas ya son BCrypt y no hace nada.

// PasswordMigrationRunner.java — se ejecuta una vez al arrancar Spring Boot @Component public class PasswordMigrationRunner implements ApplicationRunner { @Override @Transactional public void run(ApplicationArguments args) { int migrados = 0; for (Usuario usuario : usuarioRepository.findAll()) { // ¿Ya es BCrypt? Los hashes BCrypt siempre empiezan con $2 y tienen 60 chars boolean yaEsBcrypt = usuario.getPassword().startsWith("$2") && usuario.getPassword().length() == 60; if (!yaEsBcrypt) { // Reemplaza el texto plano con su hash BCrypt usuario.setPassword(passwordEncoder.encode(usuario.getPassword())); usuarioRepository.save(usuario); migrados++; } } // Ejemplos: // "admin" → "$2a$10$hKZdsMlFkdjhQ2kJMO1xyz..." (60 chars) // "1234" → "$2a$10$9xPaT0dDfGhIjKlMnOpQrs..." (60 chars) } }
